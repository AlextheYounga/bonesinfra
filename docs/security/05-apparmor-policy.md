# BonesDeploy AppArmor Policy

## Purpose

AppArmor should be the default static confinement layer for BonesDeploy services and risky workers where the host distribution supports it.
It should narrow read, write, execute, and process access beyond what systemd alone can express.

## AppArmor Should Be Enabled

AppArmor should be installed, enabled, and enforcing where supported by the distribution.

An operator or agent should check:

```bash
aa-status
systemctl status apparmor
cat /sys/module/apparmor/parameters/enabled
```

Expected result:

```text
AppArmor enabled
profiles loaded
target services in enforce mode
```

## Per-Project Profiles

Each long-running project service should have one of the following:

- a dedicated AppArmor profile
- a profile generated by a container or runtime system
- a documented reason AppArmor is not used for that service

A dedicated profile should restrict:

- readable paths
- writable paths
- executable paths
- network permissions where practical
- access to sensitive `/proc` and `/sys` paths where practical
- cross-project access

## Minimum BonesDeploy Intent

For project `app1`, the effective AppArmor intent should approximate:

```text
allow read:       /srv/deployments/app1/current/**
allow read:       /srv/deployments/app1/shared/.env
allow read/write: /srv/deployments/app1/shared/storage/**
allow read/write: /srv/deployments/app1/shared/uploads/**
allow read/write: /run/app1/**
deny:             /srv/deployments/app2/**
deny:             /srv/deployments/app3/**
deny:             /home/**
deny:             /root/**
deny:             /etc/ssh/**
deny:             /home/git/**
```

The exact syntax may vary by profile, but the access intent should stay narrow.

Note: if runtime configuration is intentionally stored under a deploy-user home path
(for example `/home/git/<project>.git/bones/*.yaml`), profile denies must not
contradict those required reads. Keep denies narrow enough to preserve required
project-scoped reads.

## Deployment and Worker Coverage

Where practical, AppArmor should also cover risky non-service processes such as:

- build workers
- deploy hooks
- plugin execution
- asset compilation jobs
- other child processes handling untrusted input

These worker profiles can be narrower than the main application profile because they usually need fewer long-lived privileges.

## Findings

The agent or operator should flag:

- app service runs unconfined
- profile is in complain mode instead of enforce mode
- service can read other projects' release trees or secrets
- service can write outside approved writable paths
- service can execute arbitrary writable files
- service can read all of `/etc/**` without need
- service can access deployment SSH keys or deploy metadata
